# 企业微信外部用户管理系统 (WeCom SCRM) 产品架构设计

本项目是一个基于企业微信“客户联系”能力的外部用户（客户/联系人）管理系统（SCRM）。

## 1. 产品概述与定位

核心业务目标是通过打通企业微信生态，实现企业外部客户资产的沉淀、运营与分析，赋能销售与运营团队，防止客户流失，并提升客户转化率。

---

## 2. 核心功能模块划分

### 2.1 客户管理 (Customer Profile & Lifecycle)
*   **客户 360 视图**：聚合客户基础信息、企微特有信息及业务自定义字段。
*   **标签体系**：双向同步企微官方标签，支持系统内自定义标签与自动打标。
*   **客户归属与分配**：清晰的客户/跟进人映射关系，支持在职转移客户。
*   **离职继承机制**：员工离职自动触发客户资产继承流程，保障企业核心资产。
*   **动态时间线**：完整记录客户从添加、进群、互动到流失的全生命周期行为。

### 2.2 客户群管理 (Group Chat Management)
*   **群资产沉淀**：同步企微客户群列表、群主及群成员变动。
*   **群活码管理**：生成和管理群活码，支持自动分流进群。
*   **群数据报表**：进退群人数趋势、群活跃度统计。

### 2.3 欢迎语管理 (Welcome Message)
*   **好友欢迎语**：新加客户触发（20秒内），支持按部分、员工、活码定制多套规则。
*   **入群欢迎语**：基于企微入群模板，实现自动发送欢迎内容。
*   **富媒体支持**：文字、图片、链接、小程序及动态变量（如客户昵称）。

### 2.4 用户运营 (Customer Operations)
*   **客户群发**：支持按标签定向群发（受企微单客日频次限制）。
*   **群发客户群**：向指定客户群广播消息。
*   **朋友圈助手**：创建企业朋友圈任务，分配给指定员工发布（每日限额 3 条）。
*   **素材库**：结构化管理话术、图片、视频等运营素材，方便快捷引用。

### 2.5 数据看板与审计 (Dashboard & Audit)
*   **多维报表**：客户数净增趋势、渠道归因分析、漏斗转化率。
*   **员工产能**：各销售/客服人员的拓客效率及流失率对比。
*   **安全审计**：敏感操作（如批量打标、客户转移、数据导出）全量留痕。

## 3. 系统架构与技术选型

### 3.1 总体架构蓝图
系统采用标准的前后端分离架构，包含以下核心层次：

1.  **接入层 (Client/UI)**：Web 管理后台（企业内部使用） + 企微工作台H5（一线员工移动端使用）。
2.  **网关与路由层 (API Gateway/Nginx)**：处理 HTTPS、防抓包、反向代理。
3.  **业务逻辑层 (Services)**：包含客户服务、运营服务、消息调度、权限管理。
4.  **防腐层 (WeCom Adapter/SDK)**：严格隔离企微 API 的变动。处理 Access Token 中控、接口限流重试、以及回调事件的解密与分发。
5.  **基础设施层 (Database & Cache)**：关系型数据库存储核心结构化数据；Redis 负责缓存与分布式控制。

### 3.2 技术栈概览

| 分层 | 技术选型 | 说明 |
| :--- | :--- | :--- |
| **前端框架** | Vue 3 / TypeScript | 主应用框架 |
| **UI 组件库** | Element Plus | 桌面端管理后台组件库 |
| **构建工具** | Vite | 高性能构建与开发服务器 |

| **通信** | WebSocket / SSE | 实时通知（如群发进度） |

## 4. 关键技术难点与应对机制

> [!IMPORTANT]
> **回调事件时效性（尤其是欢迎语）**
> *   **痛点**：企微要求在好友添加事件回调的 20 秒内调用发送欢迎语接口。
> *   **策略**：回调接口做异步解耦。接收到 XML 加密数据后，立刻返回 `success`，将解密后的 Payload 抛入高吞吐低延迟的内存队列（或 Redis Stream），由专职工作线程池高速匹配规则并调用外发 API。

> [!WARNING]
> **Access Token 的全局唯一性**
> *   **痛点**：多节点部署时若各自并发去企微刷 Token，会导致 Token 频繁失效，甚至触发封禁。
> *   **策略**：引入 Redis 作为全局锁/存储。由单一的定时任务（Job）提前（例如过期前 10 分钟）刷新 Token 并存入 Redis，所有业务模块仅从 Redis 中读取。

> [!CAUTION]
> **API 限流与可用性**
> *   **痛点**：批量导入或深夜全量对账时，易触碰企微频控拦截。
> *   **策略**：防腐层需实现漏桶或令牌桶算法，平滑请求并发；并增加针对企微 errcode: `45009` (接口调用超限) 等特定错码的退避重试 (Exponential Backoff) 机制。
